Built two systems for token-efficient testing and agent operation.

**Eval harness** (`tests/eval_harness.py`):
- Reads test cases from `tests/cases/*.yaml`
- Tests AI judgment layer directly via `ai.ask()` — not subprocess, because `dt agent` requires live serial port
- 5 test cases: basic_greeting, high_p_warning, lower_dterm, motor_test_safety, pid_query
- Optional `--judge` flag uses Haiku as LLM scorer
- Result records: 300-char preview + token estimate in JSON; full response in sidecar `{case_id}_response.txt`
- `tests/results/` gitignored

**Tool output compression** (`cli/tool_output.py`):
- Gated by `DRONETELEO_AGENT_MODE=1` env var — CLI behavior unchanged
- Strategies: `config_diff`, `config_compare`, `config_review`, `fc_context`
- Retrofitted into `analyze.py` and `agent.py` (fc_context)
- Full raw output logged to `logs/tool_output/` in agent mode

**KEY_PARAMS**: 27 params (expanded from 13). Covers PIDs, filter cutoffs (dterm/gyro lpf1 static/dyn), rates (rc_rate, srate, expo for roll/pitch/yaw), motor_protocol, failsafe_procedure, rates_type, simplified filter flags.

**Test fidelity fix**: harness compresses fc_context before injecting it (matches production). `--no-compress` flag for debugging. ~60% token reduction on fc_context per test run.

**Why:** fc_context compression was lossy for filter/rates params — tests would give false greens because uncompressed context was passed to AI, masking params the agent wouldn't see in production.
